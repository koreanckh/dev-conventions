# 원자료: 에이전트 작업 규칙 (SuperMarkit 메모리에서 추림)

출처: SuperMarkit 프로젝트의 Claude Code 메모리(feedback 타입). 프로젝트 전용 값은 걸러내고, 다른 프로젝트에도 쓸 범용 작업규칙만 모음.
import 시 참고: 아래는 대략 3~4개 주제(계획-승인 / 풀스택 병렬 / 가벼운 실행 / 커밋 전 lint)로 분해 가능. "완료 기준"은 기존 todo-workflow 규칙에 흡수하면 됨.

---

## 규칙 1: 변경 전 계획 제시 + 명시적 승인

실제 변경(파일 수정·커밋·삭제 등)에 착수하기 전에 **작업 계획을 먼저 제시하고 사용자의 명시적 승인을 받은 뒤** 실행한다. 조사·코드 읽기·질문 답변은 승인 없이 해도 됨.

- 변경 직전 요약 제시: (1) 무엇을 바꿀지, (2) 어느 파일을, (3) 어떻게 검증할지.
- 사용자가 질문하면 먼저 답하고 방향을 합의한 다음 손댄다. "고치자" 정도의 대략적 동의로 곧바로 수정+커밋까지 하지 말 것.
- 서브에이전트 dispatch도 계획 공유 후 진행.
- **중간 방법 전환도 재승인 대상:** 승인받은 계획대로 가다 막혀서 다른 접근으로 바꿔야 하면, 임의로 바꾸지 말고 (1) 무슨 문제인지 (2) 대안이 뭔지 알리고 동의를 받는다. 최초 승인이 "막히면 알아서 우회"를 뜻하지 않음.

근거: 질문에 답만 하지 않고 바로 수정+커밋해서 지적받은 사례.

---

## 규칙 2: 풀스택 작업은 계약 확정 후 병렬 dispatch

백엔드+프론트를 함께 고쳐야 하는 작업은 **spec 단계에서 API 계약(경로·쿼리 파라미터·요청/응답 JSON shape·상태코드)을 먼저 확정**한 뒤, **백엔드 담당·프론트 담당 에이전트를 한 메시지에서 동시에(fork) dispatch**해 병렬 진행한다. 프론트가 백엔드 완성을 순차로 기다리지 않게 한다.

- 백엔드 에이전트 = 컨트롤러/서비스/DTO/마이그레이션.
- 프론트 에이전트 = 합의된 계약 기준 api 클라이언트/훅/화면(필요 시 타입을 미리 정의해 계약 기준으로 작성).
- 각자 자기 repo에서 build/lint까지 → 완료 후 diff 검수 → 각 repo 커밋.

근거: 계약이 고정되면 두 쪽은 독립적이라 동시 작업이 순차보다 빠름.

## 규칙 3: 가벼운 실행 선호 (무거운 검증은 CI로)

무거운 로컬 검증 의식과 과도한 권한 프롬프트를 피하고 가볍게 진행한다.

- 무거운 로컬 검증(대용량 docker build 등)은 계획에 있어도 실행 시 **CI로 이관 가능한지 먼저 확인**. 이미지 빌드는 어차피 CI가 하므로 로컬 반복은 낭비. 빌드 주입 검증은 로컬 `pnpm build` + grep으로 대체 가능.
- 긴 자동 실행 전, 권한 프롬프트가 많겠다 싶으면 bypass permissions 모드를 안내(변경 명령은 read-only allowlist로 안 줄어듦).
- **리뷰 게이트 자체는 유지**하되, 자잘한 docs/config 태스크는 컨트롤러가 직접 처리해도 됨.

---

## 규칙 4: 커밋 전 lint/format 준수 (기존 코딩 컨벤션에 흡수)

- 코드 편집 **전에** 대상 repo의 `eslint.config.*` / `.prettierrc`를 확인하고 그 포맷 규칙을 따른다.
- husky + lint-staged pre-commit 훅을 도입한 repo에서는 훅 우회(`--no-verify`) **금지**.
- lint 설정을 안 따라 포맷 어긋난 파일을 커밋하면 리뷰/머지 때 무의미한 diff와 재정렬 커밋이 생긴다.

근거: prettier `tabWidth: 4`를 무시하고 2-space로 커밋돼 재정렬이 필요했던 사례.

---

## 규칙 5(흡수용): 완료 기준을 프로젝트마다 명시적으로 정의

to-do/이슈의 "완료" 판정 기준을 프로젝트마다 명시적으로 정한다. 예: SuperMarkit은 "`dev` 브랜치 머지 + `origin/dev` 푸시까지"를 완료로 보고, main 머지/운영 배포는 to-do 라이프사이클과 분리. 완료 처리 시 md 표기("머지 대기" 등)를 믿지 말고 실제 git 상태(`git log origin/dev..dev` 비었는지)로 검증한다.

→ todo-workflow 규칙에 "완료 기준은 프로젝트마다 정의" 패턴으로 넣으면 됨(구체 브랜치 전략은 프로젝트별).
