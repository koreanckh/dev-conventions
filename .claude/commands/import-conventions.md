# /import-conventions $ARGUMENTS

`inbox/`에 모인 다른 프로젝트 원자료(규칙 메모·config·스니펫)를 분석해서, 이 repo 패턴에 맞는 규칙 문서로 만들거나 **기존 문서를 갱신**한다. 원자료를 그대로 복붙하지 말고 이 repo 문체·구조로 다시 쓴다.

먼저 `CLAUDE.md`의 "새 규칙 추가하기" 체크리스트와 `templates/convention.template.md`를 기준으로 삼는다.

## 절차

1. **원자료 수집** — `inbox/` 아래 모든 파일을 읽는다(`inbox/README.md`, `inbox/processed/`는 제외). `$ARGUMENTS`로 경로가 주어지면 그 파일/폴더만.
2. **주제 분해** — 내용을 규칙 주제 단위로 쪼갠다(1주제=1규칙). 한 파일에 여러 규칙이 섞여 있으면 나눈다.
3. **중복 검사 (중요)** — 각 주제를 기존 `conventions/*.md`와 대조한다.
   - 이미 다루는 주제면 → 해당 문서를 **갱신**한다. 중복 문서를 새로 만들지 않는다.
   - 값이 충돌하면(예: 들여쓰기 2 vs 4) 임의로 덮지 말고 "주의 / 알려진 불일치"에 남기거나 사용자에게 확인.
   - 새 주제면 → `cp templates/convention.template.md conventions/<이름>.md`로 신규 생성.
4. **값/설정 분리** — config 같은 실값은 산문 말고 `templates/<이름>/`에 실파일로. 원자료가 실제 config면 byte-identical 스냅샷 + 폴더 `README.md`(출처·기준 날짜/버전).
5. **패턴 준수** — 각 문서: 상단 출처 줄 + 템플릿 섹션 순서 + 맨 아래 "이 규칙 적용하기".
6. **인덱스 갱신** — `README.md` "규칙 목록" 표에 행 추가/수정, `templates/CLAUDE.snippet.md`에 포인터 추가(매 작업 적용=always-on 인라인 / 상황부합=on-demand 힌트).
7. **원자료 정리** — 처리한 파일을 `inbox/processed/<YYYY-MM-DD>/`로 이동한다(삭제 금지).
8. **리포트** — 신규 생성 / 갱신 / 사용자 판단이 필요한 충돌을 목록으로 요약.

## 원칙

- 주제 분류나 중복 판단이 애매하면 **진행 전에 사용자에게 확인**한다.
- 원자료를 그대로 옮기지 않는다 — 원칙만 추리고 값은 파일로 분리한다.
- **커밋하지 않는다** (사용자가 요청할 때만).
