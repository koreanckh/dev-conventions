# 에이전트 작업 규칙

> 원본(SSOT): dev-conventions. 대상 repo로 복사할 땐 이 줄을 `> 출처: dev-conventions · 복사 YYYY-MM-DD`로 바꿔 남긴다(복사본이 낡았는지 판단용).

AI 에이전트가 이 repo/프로젝트에서 **어떻게 일하는가**에 대한 규칙. 무엇을 만드는가(도메인)나 코드를 어떻게 쓰는가(→ [코딩 컨벤션](coding-conventions.md))가 아니라, 변경을 진행하는 절차·협업 방식을 다룬다.

## 원칙

### 기본은 lightweight mode
- 범위와 완료 조건이 명확하고, 변경이 국소적이며 쉽게 되돌릴 수 있는 작업은 현재 세션에서 직접 처리한다. 별도 spec/plan 작성이나 승인 대기보다 구현과 검증에 집중한다.
- 실제 변경 전에는 무엇을 바꾸고 어떻게 검증할지 짧게 알리되, 사용자가 이미 명확하게 요청한 범위 안의 가역적 변경에는 추가 승인을 요구하지 않는다.
- 조사·코드 읽기·질문 답변은 별도 절차 없이 진행한다.
- 요구사항이 모호하거나 중요한 설계 선택이 필요하거나, 여러 컴포넌트·데이터 마이그레이션·보안/권한·되돌리기 어려운 외부 변경·장기 실행이 포함되면 full workflow 후보로 본다. 이유와 비용을 설명하고 시작 전에 한 번 승인받는다.
- 승인받은 접근에서 벗어나는 중대한 방법 전환이나 범위 확대는 문제와 대안을 설명하고 재승인받는다. 단순한 구현 세부 조정은 재승인 대상이 아니다.

### 풀스택 작업은 계약 확정 후 필요할 때 병렬 dispatch
- 백엔드+프론트를 함께 고치는 작업은 구현 전에 **API 계약(경로·쿼리 파라미터·요청/응답 JSON shape·상태코드)을 먼저 확정**한다. full workflow를 쓰는 작업이면 이 계약을 spec에 기록한다.
- 계약이 고정되고 양쪽 작업이 충분히 크고 독립적일 때만 병렬 dispatch를 제안한다. 서브에이전트 수와 리뷰 비용을 설명하고 승인받은 뒤, 백엔드 담당·프론트 담당 에이전트를 한 메시지에서 동시에 dispatch한다. 작은 변경은 컨트롤러가 직접 처리한다.
  - 백엔드 = 컨트롤러/서비스/DTO/마이그레이션.
  - 프론트 = 합의된 계약 기준 api 클라이언트/훅/화면(필요 시 타입을 미리 정의).
- 각자 자기 repo에서 범위에 맞는 검증까지 → 완료 후 컨트롤러가 diff 검수. 커밋은 사용자가 요청한 경우에만 한다.

### 작업 격리 — 워크트리는 "병합 단위"에 붙인다

워크트리가 필요한 작업은 사용자 승인이나 대상 저장소 지침을 확인한 뒤 격리된 git worktree에서 진행하고, 끝나면 병합한다. 워크트리는 **todo가 아니라 "독립적으로 병합되는 한 덩어리"** 단위로 만든다 — 에픽 todo를 통째로 워크트리 하나에 담으면 오래 사는 브랜치가 main에서 멀어져 병합이 어려워진다. 추적 단위(todo/plan) 정의는 [TODO 관리](todo-workflow.md)의 "spec/plan 연계" 참고.

- **작고 원자적인 todo** (한 브랜치·한 병합으로 끝): todo 하나 = 워크트리 하나. 브랜치 `todo/NNN-<slug>`.
- **에픽 todo** (도구·화면 여러 개를 며칠·여러 세션에 걸쳐 담는 우산): 에픽은 우산으로만 두고, 워크트리는 그 밑의 **plan(또는 자식 todo) 단위**로 만든다. 한 에픽이 워크트리 여러 개로 갈라지는 게 정상이다(병렬 세션 격리).
- **사소·기계적 작업:** 워크트리 없이 처리한다.
- **브랜치명엔 항상 todo 번호를 실어** 추적성을 유지한다(`todo/NNN-<slug>`, 또는 기능 기반 이름을 쓰더라도 번호를 포함).
- **수명:** 워크트리는 todo가 `진행`으로 바뀔 때 만들고 병합 즉시 제거한다. `대기` 항목엔 만들지 않는다(워크트리 개수 폭발 방지).
- 병합 대상·완료 기준은 프로젝트마다 다르다 → [TODO 관리](todo-workflow.md)의 "완료 기준" 참고.

### 가벼운 실행 (무거운 검증은 CI로)
- 검증은 변경 위험에 비례한다. 작은 변경은 관련 테스트·타입 검사·제한된 빌드만 실행하고, 전체 검증은 영향 범위가 넓거나 대상 저장소 지침이 요구할 때 실행한다.
- 무거운 로컬 검증(대용량 docker build 등)은 계획에 있어도 실행 전 **CI로 이관 가능한지 먼저 확인**한다. 이미지 빌드는 어차피 CI가 하므로 로컬 반복은 낭비. 빌드 주입 검증은 로컬 `pnpm build` + grep으로 대체 가능한 경우가 많다.
- 긴 자동 실행 전 권한 프롬프트가 많겠다 싶으면 bypass permissions 모드를 안내한다(변경 명령은 read-only allowlist로 줄지 않음).
- 리뷰 강도도 위험에 비례한다. 자잘한 docs/config 태스크는 컨트롤러가 직접 diff를 확인하고, 별도 리뷰 에이전트는 중요한 통합·설계·보안 변경에만 사용한다.

## 주의

- "가벼운 실행"이 검증 생략을 뜻하지 않는다. 변경 범위에 맞는 최신 증거를 남기되, 품질 이득이 없는 계획·리뷰 반복을 피한다.
- 완료 판정 기준은 프로젝트마다 다르다 → [TODO 관리](todo-workflow.md)의 "완료 기준" 참고.

## 이 규칙 적용하기

1. 이 문서를 대상 repo `docs/conventions/agent-workflow.md`로 복사하고 상단 출처 줄을 복사일로 채운다.
2. `AGENTS.md`의 `## 공통 규칙`에 포인터를 추가한다(전체 블록은 `templates/AGENTS.snippet.md`). always-on 핵심 규칙 + 포인터 한 줄:
   ```markdown
   - 작업 방식의 기본값은 lightweight mode다. 범위와 완료 조건이 명확하고 국소적이며 쉽게 되돌릴 수 있는 변경은 별도 spec/plan·워크트리·서브에이전트 없이 직접 처리한다.
   - pre-commit 훅 우회(`--no-verify`) 금지.
   - 풀스택 병렬 진행·가벼운 실행 등 작업 방식: docs/conventions/agent-workflow.md
   ```
