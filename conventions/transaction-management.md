# 트랜잭션 관리

> 원본(SSOT): dev-conventions. 대상 repo로 복사할 땐 이 줄을 `> 출처: dev-conventions · 복사 YYYY-MM-DD`로 바꿔 남긴다(복사본이 낡았는지 판단용).

데이터 정합성을 지키기 위한 **트랜잭션 경계·롤백·부수효과·동시성** 규칙. 언어/프레임워크에 무관한 원칙만 담는다. 구체적 API(`@Transactional`, `BEGIN/COMMIT`, `tx.Rollback()` 등)는 스택마다 다르므로 맨 아래 "구현 매핑"으로만 연결한다.

## 원칙

- **한 비즈니스 연산 = 한 트랜잭션.** 하나의 논리적 작업에 속한 여러 쓰기는 전부 커밋되거나 전부 롤백된다. "절반만 반영된 상태"를 만들 수 있는 코드 경로를 남기지 않는다.
- **트랜잭션은 짧게.** 경계 안에는 DB 읽기/쓰기만 둔다. 밖으로 빼는 것: 외부 API/RPC 호출, 메시지·이벤트 발행, 이메일/푸시 알림, 파일 업로드, 사용자 입력 대기, 무거운 순수 연산. 트랜잭션이 열려 있는 동안 커넥션과 락을 잡고 있다는 걸 항상 의식한다.
- **부수효과는 커밋 이후에.** 이벤트 발행·알림·외부 호출은 커밋이 확정된 뒤 실행한다. 커밋 전에 내보내면 롤백돼도 이미 나간 유령 이벤트가 남는다. after-commit 훅 또는 outbox 패턴(같은 트랜잭션에 아웃박스 행을 쓰고 커밋 후 별도로 발송)을 쓴다.
- **실패하면 롤백이 기본.** 예외를 삼켜 부분 커밋을 만들지 않는다. 트랜잭션 안에서 잡아야 하는 예외라면, 잡은 뒤 트랜잭션을 rollback-only로 두거나 다시 던져 경계까지 전파시킨다.
- **경계는 명시적으로 한 곳에서 연다.** 트랜잭션 시작/커밋 지점을 여러 레이어에 흩뿌리지 말고 서비스/유스케이스 레이어 한 곳에서 시작한다. 하위 함수는 "이미 열린 트랜잭션에 참여"할 뿐, 스스로 커밋하지 않는다.
- **읽고 나서 쓰면 동시성을 가정한다.** read-modify-write는 lost update가 생긴다. 낙관적 락(버전 컬럼)이나 조건부 업데이트(`UPDATE ... WHERE version = ?`)로 막고, 정말 필요한 구간만 비관적 락(`SELECT ... FOR UPDATE`)을 쓴다.
- **락 순서를 일관되게** 잡아 데드락을 피한다. 한 트랜잭션에서 여러 자원을 잠글 때는 코드 어디서든 **항상 같은 순서**로 잠근다.
- **멱등성을 확보한다.** 재시도·중복 요청에도 결과가 한 번만 반영되게 한다. 유니크 제약 + upsert, 또는 idempotency key로 중복을 흡수한다. 네트워크가 끼면 "성공했는지 모르는 상태"의 재시도는 항상 일어난다고 가정한다.
- **격리 수준은 기본값을 유지하고, 올릴 땐 근거를 남긴다.** 대부분은 DB/프레임워크 기본값으로 충분하다. 팬텀·논리적 정합성이 꼭 필요한 구간만 상향하고 왜 올렸는지 주석을 단다.
- **여러 저장소에 걸친 원자성(분산 트랜잭션/2PC)은 지양한다.** 경계를 하나의 저장소 안으로 좁히는 설계를 먼저 시도하고, 불가피하면 saga + 보상 트랜잭션(compensation)으로 최종 일관성을 맞춘다.

## 안티패턴 / 주의

- **트랜잭션 안 외부 호출** — HTTP/RPC를 트랜잭션 안에서 부르면 그 지연만큼 커넥션·락을 점유하고, 상대가 느리거나 타임아웃 나면 롤백 처리가 애매해진다.
- **락/트랜잭션 없는 read-modify-write** — 조회한 값으로 계산해 다시 쓰면 동시 요청끼리 서로의 갱신을 덮어쓴다(lost update).
- **커밋 전 이벤트 발행** — 롤백돼도 이벤트는 이미 나가 소비자가 없는 데이터를 참조한다.
- **예외를 잡고 그냥 진행** — ORM에 따라 트랜잭션이 이미 rollback-only로 마킹돼, 뒤에서 커밋을 시도하면 되레 실패한다.
- **요청 전체를 한 트랜잭션으로 감싸기** — 요청 처리 내내 트랜잭션이 열려 있어 락을 오래 잡는다. 실제 쓰기 구간만 감싼다.
- **긴 배치를 한 트랜잭션으로** — 언두 로그/락이 팽창한다. 청크 단위로 나눠 커밋하고, 중단 시 재개 가능하게(멱등) 설계한다.
- **전파(propagation) 오해** — "기존 트랜잭션에 참여"와 "새 트랜잭션 시작"은 다르다. 중첩 호출에서 어느 쪽인지 모르고 쓰면 의도치 않은 커밋/롤백 범위가 생긴다.

## 구현 매핑

개념은 동일하고 표현만 스택마다 다르다. 대상 repo에서 쓰는 것에 맞춰 참고한다.

- **Spring / JPA·MyBatis** — `@Transactional`(경계·전파·격리), `@Version`(낙관적 락), `@TransactionalEventListener(phase = AFTER_COMMIT)` 또는 `TransactionSynchronization`(커밋 후 부수효과).
- **Node (TypeORM/Prisma/Knex)** — `dataSource.transaction(cb)` / `prisma.$transaction([...])` / `knex.transaction(cb)`. 부수효과는 콜백 밖(커밋 후)에서.
- **Django** — `with transaction.atomic():`, 커밋 후 부수효과는 `transaction.on_commit(cb)`.
- **Go (database/sql)** — `tx, _ := db.BeginTx(...)` + `defer tx.Rollback()` 후 성공 시 `tx.Commit()`.
- **Raw SQL** — `BEGIN` / `COMMIT` / `ROLLBACK`, 잠금은 `SELECT ... FOR UPDATE`.
